# スコープ外指摘の切り出し（spinoff）

## 役割

`/review` 等でスコープ外として棚上げされた指摘を、その場で議論・要件化せずに独立 feature の
"種"文書 `.specs/<新feature>/seed.md` として切り出す。切り出しは非対話を基本とし、対話は
「どの指摘を・分割/まとめ・切り出し先名」というルーティング確認の1ステップに限定する。

**位置づけ**: seed.md は「機能要望」であって requirements.md ではない。切り出し先 feature で
`/spec` に渡すと、質問しながら requirements.md に詳細化され、その後 design → tasks → impl と
正規フローに乗る。よって本スキルは実装手順・受け入れ条件・EARS 要件を先回りして書かない
（spec→design→tasks が生成する）。書くのは「何を・なぜ・どこまで」に絞る。

**非対話原則**:
- 指摘の中身について要件詳細化のための質問はしない。
- 不足している情報は seed.md 内に「TODO: /spec で詳細化」と明記するだけで、質問で埋めない。
- EARS 形式の `requirements.md` は生成しない（詳細化は `/spec` の責務）。
- 対話するのはルーティング確認（どれを・分割orまとめ・スラッグ名）のみ。これは中身の議論では
  なく切り出し先の振り分けであり、非対話原則の例外として許容する。

## 引数

- 第1引数: 元 feature 名（`.specs/<feature>/review.md` からスコープ外指摘を拾う）
- プロンプトで指摘内容を直接渡すこともできる（review.md と併用可）

## 進め方

### Step 1: 入力の特定

- `$ARGUMENTS[0]`（元 feature）があれば `test -f .specs/<feature>/review.md` を確認し、あれば読む。
- プロンプトで指摘が直接渡されていれば、それも切り出し対象の候補に含める。
- 元 feature が未指定かつプロンプトの指摘も無ければ「使い方: /spinoff <feature>」を表示して終了する。
- review.md が存在せず、プロンプトの指摘も無ければ「切り出せるスコープ外指摘が無い」旨を伝えて終了する。

**完了ゲート:** 切り出し対象の候補（review.md 由来 / プロンプト由来）が確定したか。

### Step 2: 候補の抽出

- review.md から指摘（`### CodeRabbit の指摘` / `### 推奨対応` 等の記録項目）を候補として列挙する。
- review.md にはスコープ外を示す公式マーカーが無いため、ここで機械的に絞り込まず、Step 3 の
  確認でユーザーに対象を選ばせる。
- 指摘の中身についての質問はしない。要件詳細化はここでは行わない。

### Step 3: ルーティング確認（唯一の対話点）

`AskUserQuestion` で以下をまとめて確認する。これは振り分けの確認であり、指摘内容の議論・
要件化ではない:

1. 切り出す指摘（複数候補があるとき、どれを切り出すか）
2. 「独立した複数 seed に分割」か「1つの seed にまとめる」か
3. 各切り出し先 feature 名（スラッグ、kebab-case）案。指摘見出しから提案する

指摘が1件のみのときはこのステップを省略し、そのまま Step 4 の生成に進む。

良いスラッグ名が作れないときも質問で深掘りせず、無難なスラッグを提示してユーザー承認に委ねる
（非対話原則の維持）。

**完了ゲート:** 切り出し対象・分割方針・切り出し先 feature 名が確定したか（1件のみなら省略で確定）。

### Step 4: seed 生成

確定した切り出し先ごとに以下を行う:

1. スラッグが安全か検証する。**kebab-case の単一セグメント**のみを許可し、パス区切り（`/`）・
   `..`・絶対パスを含むものは弾く。書き込み先が `.specs/` 配下に収まることを確認する。逸脱して
   いれば生成せず、Step 3 に戻って別スラッグを承認してもらう。
2. `test -d .specs/<新feature>` で衝突チェックする。既存なら**上書きせず中断**し、別名の指定を促す。
3. 衝突が無ければ `.specs/<新feature>/seed.md` を下記フォーマットで Write する。既知情報のみ記述し、
   不足は「TODO: /spec で詳細化」と明記する。
4. `requirements.md` は生成しない。

複数 seed を一括生成する場合、1つでも衝突があれば**衝突分だけ中断**し、他は生成する（部分成功を報告）。

**完了ゲート:** 衝突の無い切り出し先すべてに seed.md を Write したか（衝突分は中断理由を把握したか）。

### Step 5: 元 feature には何も書かない

元 feature の `review.md` などに切り出しの追跡記録は残さない。これは明示的な設計であり、
「切り出し済み」のマーキングや戻り値を書き込む処理は行わない。

### Step 6: 導線提示

生成した各 `seed.md` のフルパスと、次に `/spec <新feature>` を実行する導線を提示する。
`/spec` は自動起動しない。末尾の「次ステップ提示」の定型ブロックで示す。

## seed.md フォーマット（統一テンプレート）

`/spinoff` と `/handoff` の双方が出力する共通フォーマット。EARS 要件ではなく「機能要望」の
粒度で書く。実装手順・受け入れ条件は書かない（`/spec`→`design`→`tasks` が生成する）。

```markdown
# <タイトル: 何をしたい変更か 1 行で>

## 背景・目的
なぜこの変更が要るか。専門用語・内部用語は本文で説明する。

## やりたいこと
何を実現したいか。挙動・体験のレベルで書く（実装方法は指定しない）。

## 対象範囲・既知の手がかり
実コードで確認した対象ファイルや該当箇所（参考情報。詳細化の起点）。
不足している情報は「TODO: /spec で詳細化」と明記し、質問で埋めない。

## 制約・前提
外せない条件、やってほしくないこと、既存仕様との整合など。

## 拾い方
この種文書は未着手。`/spec <新feature>` を実行して requirements 詳細化から始める。
```

## 完了条件

切り出し対象の feature ごとに `.specs/<新feature>/seed.md` を Write し（衝突分は中断・報告）、
元 feature 側には追跡記録を残さず、`/spec <新feature>` への導線を報告したら完了。

## エラー処理

- 元 feature 未指定かつプロンプトの指摘も無い → 「使い方: /spinoff <feature>」を表示して終了
- review.md も無くプロンプト指摘も無い → 「切り出せるスコープ外指摘が無い」旨を伝えて終了
- 切り出し先 feature ディレクトリが既に存在する → 上書きせず、その旨を伝えて別名の指定を促す
  （他の切り出し先は生成を続行する）

## 次ステップ提示

seed 生成が完了したら、次の定型ブロックを**コードフェンスで囲まず**プレーンテキストで出力し、
`/spec` は自動起動せずユーザーの実行を待って終了する。

```
✅ 切り出し完了
📄 .specs/<新feature1>/seed.md
📄 .specs/<新feature2>/seed.md（複数生成した場合はすべて列挙）
OK：/spec <新feature>
```
