# 他 PJ への修正依頼（handoff）

## 役割

今のセッションから直接編集できない別 PJ に対し、修正依頼を `.specs/` 経由で受け渡す。
相手 PJ の `.specs/<機能名>/spec.md` に、相手が拾って機械的に実装できる粒度の依頼を書く。

**前提**: ハーネスは作業ディレクトリ外のファイル編集をブロックする。例外は各 PJ の `.specs/`
配下のみ。よって本体ファイルの編集は試みず、必ず相手 PJ の `.specs/` に書く。

## 引数

- 第1引数: 依頼先 PJ のパス、または CLAUDE.md のプロジェクト略称表にある略称
- 第2引数: 機能名（`.specs/<機能名>/` のディレクトリ名になる。kebab-case）

どちらも省略時は人間に聞く。

## 進め方

### Step 1: 依頼先 PJ を特定する

引数のパス/略称から依頼先 PJ のルートを確定する。略称なら CLAUDE.md の対応表で解決する。
確定したら `.specs/` に書けることを確認する（存在しなければ作成方針を人間に確認）:

```
test -d <依頼先PJ>/.specs && echo writable || echo "(.specs なし)"
```

**完了ゲート:** 依頼先 PJ ルートと `.specs/` の書き込み可否が確定したか。

### Step 2: 依頼内容を固める

人間から修正内容を受け取り、相手が手元の文脈ゼロで実装できる粒度に落とす。CLAUDE.md の
「PR ディスクリプションの書き方」と同じ精神で、**初見の人が 1 度で読み通せる**こと。

依頼先のコードを Read して、対象ファイルの現状（パス・該当箇所）を確認してから書く。
ドキュメントや記憶を鵜呑みにせず実コードで裏取りする（CLAUDE.md「ドキュメントを鵜呑みにしない」）。

**完了ゲート:** 対象ファイルの現状を実コードで確認し、変更内容が具体化したか。

### Step 3: 依頼 spec を書く

`<依頼先PJ>/.specs/<機能名>/spec.md` に Write する。テンプレート:

```markdown
# <タイトル: 何をする依頼か 1 行で>

## 背景・目的
なぜこの変更が要るか。専門用語・内部用語は本文で説明する。

## 対象ファイル
- <絶対パスまたは PJ ルートからの相対パス>

## 変更内容
ファイルごとに、どこを・どう変えるか。before/after かコードブロックで具体的に。
新規追加か既存編集かを明示する。

## 実装手順
1. <依頼先PJ> で claude を起動する
2. 本 spec に沿って編集する
3. 動作確認（テスト/lint 等、PJ の流儀に従う）
4. `/commit` でコミットする

## 受け入れ条件
- [ ] <この依頼が満たされたと言える条件>
```

依頼元のセッションでは編集できない旨と、相手 PJ で起動し直して拾う旨を spec に明記する。

**完了ゲート:** spec を Write し、相手が単独で実装できる内容になっているか。

### Step 4: 報告する

書いた spec のフルパスと、相手 PJ での拾い方（`cd <依頼先PJ> && claude` → spec を読んで実装 →
`/commit`）を人間に報告する。

## 完了条件

依頼先 PJ の `.specs/<機能名>/spec.md` に実装可能な粒度の修正依頼を書き、拾い方を報告したら完了。

## エラー処理

- 依頼先 PJ が特定できない / 略称が対応表にない → 人間に確認する
- `.specs/` が存在しない → 作ってよいか人間に確認する（穴あけ対象か不明なため）
- 本体ファイルの編集がブロックされた → 想定どおり。編集は諦め `.specs/` への Write に徹する
