# PRコメント対応

## 役割
自分のPRに付いた未解決のレビューコメントに対応し、各コメントに返信する。スレッドの「解決済み」化はしない（解決はレビュアーに委ねる）。

## 用語（前提）
用語は `claude/CLAUDE.md`「用語集」に従う。**フェーズ = 大タスク = ブランチ = 1 PR**。複数フェーズは stacked PR。stacked のときは **feature の全 PR を横断**して未解決コメントを集め、**指摘の付いた PR のブランチ（フェーズ）で修正**する。下位フェーズを直したら stack を rebase 伝播して上位ブランチに反映する（自走で安全に伝播できないと判断したら報告して停止）。

## 自走モード（`auto` 引数）
`$ARGUMENTS` に `auto` が含まれる場合：
- **CodeRabbit（author: `coderabbitai[bot]`）のコメントのみ**を対象にする
- Step 3 の「対応方針を人間と議論する」は自動判断に読み替える（明確な指摘は対応、CodeRabbit の nit / optional は対応 or 理由付き返信を自分で判断）
- **人間のレビューコメントが付いていたら自走を止めて報告する**（人間コメントは自動対応しない）

引数なしの単体起動では従来どおり全コメントを対象に人間と議論する。

## 進め方
各 Step は順番に実行する。完了ゲートを通過するまで次へ進まない。

1. 対象PRを特定する
2. 未解決コメントを集める（インライン + 会話 + レビュー総評）
3. コメントごとに対応方針を人間と議論する
4. 対応が必要なものを実装する
5. 各コメントに返信する
6. 結果を報告する

### Step 1: 対象PRを特定する（単一 / stacked）

現在のブランチから判定する：
- 現在のブランチが `<feature>-pN`（末尾 `-p` + 数字）→ **stacked の可能性**。feature 名を求めてフェーズブランチの PR を番号順に列挙する：
  ```
  for b in $(git branch --list "<feature>-p*" | sed 's/^[ *]*//' | sort -t p -k2 -n); do
    gh pr list --head "$b" --json number,title,url --jq '.[]'
  done
  ```
  2 件以上ヒットしたら **stacked モード**（Step 2 以降を全 PR 横断で行い、各指摘はその PR のブランチで対応する）。
- それ以外は単一 PR：
  ```
  gh pr view --json number,title,url
  ```
検出できれば人間に確認する（`auto` は確認不要）。検出できなければPR番号を聞く。

**完了ゲート:** 対象PR（stacked なら全 PR）を確定したか。

### Step 2: 未解決コメントを集める

インラインのコメントだけ見るとレビュー総評や会話コメントを取りこぼすため、**3種類すべて**を集める。

**① インラインのレビューコメント（返信先 id つき）**
```
gh api repos/{owner}/{repo}/pulls/<PR番号>/comments \
  --jq '[.[] | select(.in_reply_to_id == null)] | sort_by(.path, .line)'
```
`select(.in_reply_to_id == null)` でルートコメントのみ取得する。`line != null` でフィルタするとファイル全体へのコメント（line: null）が漏れるので使わない。

**② 解決済みスレッドの除外**
①には解決済みも含まれる。未解決スレッドを reviewThreads で取得し、`path` + `line` + 本文で①と突き合わせて、解決済みのものを対象から外す：
```
gh pr view <PR番号> --json reviewThreads \
  --jq '[.reviewThreads[] | select(.isResolved == false) | {path, line, body: .comments[0].body}]'
```

**③ PR会話タブの一般コメント**
```
gh api repos/{owner}/{repo}/issues/<PR番号>/comments --jq '.[] | {user: .user.login, body}'
```

**④ レビューの総評**（Approve / Request changes 時の本文）
```
gh pr view <PR番号> --json reviews --jq '.reviews[] | select(.body != "") | {author: .author.login, state, body}'
```

**完了ゲート:** ①③④を取得し、②で解決済みインラインを除外した、未対応コメントの一覧ができたか。

### Step 3: 対応方針を議論する

コメントごとに「対応する／対応しない（理由つき）」を人間と決める。

**完了ゲート:** すべてのコメントに方針が付いたか。

### Step 3.5: 対応しない項目を tasks.md に追記する

「対応しない」と決めたコメントが1件以上ある場合、`.specs/` 配下の `tasks.md` に追記する。

**tasks.md の場所を特定する:**
```
find . -path './.git' -prune -o -name 'tasks.md' -print | grep '\.specs/'
```
複数見つかった場合は PR のブランチ名や機能名から最も近いものを選ぶ。見つからなければ人間に確認する。

**追記フォーマット:**
```markdown
## 残対応（PR #<番号> で見送り）

- [ ] <コメントの要点> <!-- <対応しない理由> -->
- [ ] ...
```
既に「残対応」セクションがあれば、その末尾に項目を追加する。新規セクションは `tasks.md` の末尾に追加する。

**完了ゲート:** 「対応しない」項目がすべて tasks.md に記録されたか。

### Step 4: 実装する

対応すると決めたものを実装する。

**stacked の場合**: 各指摘は**それが付いた PR のブランチ（フェーズ）で修正・コミット**する。下位フェーズを直したら、上位フェーズのブランチへ stack を rebase 伝播して再 push する（`git rebase --onto` 等）。伝播でコンフリクトが多発する・自走で安全に行えないと判断したら、対応済みぶんを報告して停止する。

**完了ゲート:** 方針どおりに実装し、stacked なら該当フェーズのブランチに反映（＋上位へ伝播）したか。

### Step 5: 各コメントに返信する

対応しないと決めたコメントにも、理由を添えて返信する。

**インラインコメントへの返信:**
```
gh api repos/{owner}/{repo}/pulls/<PR番号>/comments/<comment_id>/replies \
  -X POST -f body="<返信内容>"
```

**会話コメント・レビュー総評への返信:** PRコメントで返す。
```
gh pr comment <PR番号> --body "<返信内容>"
```

スレッドの解決済み化はしない（レビュアーが解決する運用）。

**完了ゲート:** 対応・非対応を問わず、すべてのコメントに返信したか。

### Step 5.5: 再発しそうなレビュー指摘ならmemoryに残す

コメントの中に **レビュアーの恒常的な好み・PJ の規約**（例: この PJ では必ず〜の命名を求められる、
特定の書き方は毎回 nit が付く、テストの置き場所が決まっている）と判断できるものがあれば、
memory に1件保存する。

- 保存先は通常のmemory（`~/.claude/projects/<cwd>/memory/`、1メモリ=1ファイル、`MEMORY.md` に
  インデックス1行追記）。type は `feedback`。body には **Why** と **How to apply** を書く
- **保存しないもの**: その PR 限りの個別指摘、git log / diff で追える対応内容。既に同趣旨の
  メモリがあれば新規作成せず更新する
- 保存したら「<要点> をmemoryに記録しました」と一言報告する

**完了ゲート:** 再発しそうな指摘の有無を判断し、あれば memory に記録したか。

### Step 6: 報告する

対応・返信の結果を人間に報告し、末尾の「次ステップ提示」の定型ブロックで次の手順を提示する。
「対応しない」項目がある場合は、追記した tasks.md のパスと項目数も合わせて報告する。

## 完了条件
未対応コメントすべてへの対応・返信が完了し、「対応しない」項目が tasks.md に記録され、
再発しそうなレビュー指摘があれば memory に記録したら完了。

## 次ステップ提示
単体起動で完了したら、対応内容に応じて次の分岐ブロックを**コードフェンスで囲まず**プレーンテキストで出力し、次スキルは自動起動せずユーザーの実行を待って終了する。

```
────────────────────────────────
✅ コメント対応・返信完了
📄 .specs/<feature>/tasks.md（「対応しない」項目を追記した場合）
▶ 次のステップ
   ・修正を push した → /watch-ci で再監視
   ・追加の修正なし → マージ / Ready for review を判断（停止点）
   理由: 再 push の有無で再監視に戻るか停止点かが決まる
────────────────────────────────
```

📄 の行は tasks.md へ追記しなかった場合は省略する。自律モード（起動引数に `auto` を含む）では上記ブロックを出さず、確定した遷移先を 1 行の簡易ログだけ残す（例: `次: /watch-ci`）。次スキルの起動は呼び出し元（orchestrator）が行う。
