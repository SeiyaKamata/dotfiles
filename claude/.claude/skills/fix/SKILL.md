# テスト修正スキル

## 役割
`/test` から渡された失敗情報をもとに、**テストを通すための最小限の修正**を行う。
新機能の追加や設計変更は行わない。それが必要な場合は `/impl` または `/design` に戻す。

## 自走モード（`auto` 引数）
`$ARGUMENTS` に `auto` を含む場合は自律モード（ブロック抑制）とする。Step 3 の次ステップ提示で定型ブロックを出さず、1 行の簡易ログのみ残す。引数なしの単体起動では定型ブロックを表示する。

## 入力
`/test` から渡される失敗情報：
- 失敗したテスト名
- エラーメッセージ
- 関連ファイル（推測）

## 進め方

### Step 1: 根本原因の特定
エラーメッセージと関連ファイルを読み、失敗の原因を特定する。

原因を以下に分類する：
- **実装バグ** → このスキルで修正
- **テストの期待値ずれ** → このスキルで修正（仕様と照合してから）
- **設計の問題** → `/design` に戻すべき。ユーザーに報告して止まる
- **環境・依存関係の問題** → ユーザーに報告して止まる

### Step 2: 修正
特定した原因に対して最小限の変更を加える。

### Step 3: テスト再実行
修正後にテストを再実行して確認する。

- PASS → 末尾の「次ステップ提示」の定型ブロックで次の手順（`/test` の再実行）を提示する
- FAIL（別の原因）→ Step 1 に戻る（最大3回まで）
- 3回試みても FAIL → ユーザーに状況を報告して指示を仰ぐ

## 完了条件
テストが PASS になったら完了。`/test` を起動して再実行する。

## 次ステップ提示
単体起動で完了したら、次の定型ブロックを**コードフェンスで囲まず**プレーンテキストで出力し、次スキルは自動起動せずユーザーの実行を待って終了する。

```
────────────────────────────────
✅ 修正してテスト再 PASS
▶ 次のステップ
   /test
   理由: 修正が入ったのでテストを通しで再実行し確定させる
────────────────────────────────
```

自律モード（起動引数に `auto` を含む）では上記ブロックを出さず、遷移先を 1 行の簡易ログだけ残す（例: `次: /test`）。次スキルの起動は呼び出し元（orchestrator）が行う。
