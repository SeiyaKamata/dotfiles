# コミット

## 役割
変更内容を確認してコミットプランを人間に提示し、承認後に分割コミットを実行する。

## 自走モード（`auto` 引数）
`$ARGUMENTS` に `auto` が含まれる場合、Step 4 のプラン承認を自動承認として扱い、プランを提示したらそのまま Step 5 を実行する（プラン内容はログとして残す）。引数なしの単体起動では従来どおり承認を待つ。

## 進め方
各 Step は順番に実行する。末尾の完了ゲートを通過するまで次へ進まない。

1. ブランチが安全か確認する（デフォルト／マージ済みなら新ブランチを切る）
2. 変更ファイルを確認する
3. コミットプランを組み立てる
4. プランを提示して承認を得る
5. プラン順にステージング＆コミットする
6. push または PR 作成を提案する

### Step 1: ブランチチェック

現在のブランチ名とデフォルトブランチ名を取得する：
```
CURRENT=$(git branch --show-current)
DEFAULT=$(git remote show origin | sed -n 's/.*HEAD branch: //p')
```

判定：
- `CURRENT` が空（detached HEAD）→ 警告へ
- `CURRENT` が `DEFAULT` と一致 → 警告へ
- リモートでマージ済み（下記がマッチ）→ 警告へ
  ```
  git fetch origin
  git branch -r --merged "origin/$DEFAULT" | sed 's/^[ *]*//' | grep -xF "origin/$CURRENT"
  ```
- いずれも該当しなければ Step 2 へ

**警告文:**
> 現在のブランチ `<CURRENT>` は <デフォルトブランチ／マージ済み／detached> の状態です。新しいブランチを切ってからコミットしますか？

- 「はい」→ デフォルトブランチを最新化してから新ブランチを作成し Step 2 へ
  ```
  git checkout "$DEFAULT" && git pull
  git checkout -b <新ブランチ名>
  ```
  ブランチ名は変更内容から `<type>/<短い説明>`（例: `fix/login-redirect`）の形で提案する。
- 「このまま進める」→ Step 2 へ

**完了ゲート:** ブランチ状態を判定し、必要なら新ブランチを切ったか。

### Step 2: 変更ファイルを確認する

```
git status
git diff
```

**完了ゲート:** 変更内容を把握し、生成物・秘密情報・無関係ファイルなど **コミットすべきでないものが混ざっていないか** を確認したか。

### Step 3: コミットプランを組み立てる

変更を「作業の流れが伝わる単位」に分割する。コミット順は依存・論理の流れを意識する（例: リネーム → 内容修正 → 新規追加）。各項目に **対象ファイルを必ず列挙する**（Step 5 のステージングで使う）。

```
## コミットプラン

1. <type>(<scope>): <subject>
   対象ファイル: foo.md, bar.md
   内容: ...

2. <type>(<scope>): <subject>
   対象ファイル: baz.md
   内容: ...
```

**完了ゲート:** すべての変更ファイルがいずれかの項目に割り当てられ、コミットすべきでないファイルは除外されているか。

### Step 4: プランを提示して承認を得る

プラン全文（対象ファイル含む）を提示し、承認を得てから Step 5 に移る。

**完了ゲート:** 人間の承認を得たか。

### Step 5: ステージング＆コミットを実行する

プランの項目ごとに、**その項目の対象ファイルだけを `git add` してからコミットする**。これを順番に繰り返す。

```
git add foo.md bar.md
git commit -m "$(cat <<'EOF'
<type>(<scope>): <subject>

<body>

Co-Authored-By: <ハーネス指定のモデル行>
EOF
)"
```

**コミットメッセージ（Conventional Commits）:**

- **type:** `feat` / `fix` / `refactor` / `docs` / `test` / `chore` / `perf` / `ci` / `build` / `revert`
- **subject:** 変更内容を一言で（命令形・現在形）
- **body:** レビュアーが背景・意図・影響範囲を理解できるように書く（なぜ必要か／何をどう変えたか／影響範囲・注意点）

**注意:**
- メッセージは HEREDOC で渡す。変数展開を避けるため本文は `<<'EOF'`（literal mode）を使う
- 末尾に `Co-Authored-By` 行を必ず付与する。モデル行はハーネス（環境）が指定する形式に従い、**スキル内に固定のモデル名を書かない**（ズレ防止）

**完了ゲート:** プランの全項目を、対象ファイル単位でコミットしたか。

### Step 6: push または PR 作成を提案する

コミット完了後に確認する：

> push しますか？／ PR を作成しますか？（/create-pr を使います）

push 時にリモートのデフォルトブランチへ直接 push しようとしていないか確認する。回答に応じて `git push` または `/create-pr` を実行し、どちらも不要なら終了する。
