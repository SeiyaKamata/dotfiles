# コミット

## 役割
変更内容を確認してコミットプランを人間に提示し、承認後にコミットする。

## 進め方
1. 現在のブランチがリモートでマージ済みか確認する
2. 変更ファイルを確認する（`git status`, `git diff`）
3. 変更を作業の流れが伝わる単位に分割し、コミットプランを組み立てる
4. コミットプランを人間に提示し、承認を得る
5. 承認後に順番にコミットを実行する

### Step 1: マージ済みブランチチェック

現在のブランチ名を取得し、リモートでマージ済みか確認する：
```
git branch --show-current
git branch -r --merged origin/main | grep <現在のブランチ名>
```

マージ済みと判明した場合（またはデフォルトブランチ上にいる場合）、ユーザーに警告して確認する：

> 現在のブランチ `<ブランチ名>` はリモートでマージ済みです。新しいブランチを切ってからコミットしますか？

- 「はい」→ ベースブランチ（origin/main等）を最新化してから新ブランチを作成し、コミットに進む
  ```
  git fetch origin
  git checkout main && git pull
  git checkout -b <新ブランチ名>
  ```
  ブランチ名はユーザーに提案するか、変更内容から自動命名する。
- 「このまま進める」→ そのままコミットに進む

## コミットプランの形式

コミットは「なぜその順番か」が伝わるように並べる。
例: リネーム → 内容修正 → 新規追加、のように依存関係・論理的な流れを意識する。

```
## コミットプラン

1. <type>(<scope>): <subject>
   対象ファイル: foo.md, bar.md
   内容: ...

2. <type>(<scope>): <subject>
   対象ファイル: baz.md
   内容: ...
```

承認を得てから実行に移る。

## コミットメッセージの規則
Conventional Commits形式を使用する。

```
<type>(<scope>): <subject>

<body>

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

**type一覧:**
- `feat`: 新機能
- `fix`: バグ修正
- `refactor`: リファクタリング
- `docs`: ドキュメント
- `test`: テスト
- `chore`: ビルド・設定変更

**subject:** 変更内容を一言で（命令形・現在形）

**body:** レビュアーが背景・意図・影響範囲を理解できるように記述する。以下を意識する：
- なぜこの変更が必要だったか（背景・問題）
- 何をどう変えたか（具体的な変更内容）
- 影響範囲や注意点があれば記載

**注意:**
- コミットメッセージはHEREDOCで渡す（特殊文字のエスケープを防ぐため）
- `Co-Authored-By` 行を必ず末尾に付与する

## 完了条件
コミット完了後に以下を聞く：

> PRを作成しますか？（/create-pr を使います）
> すでにPRがある場合はpushしますか？

回答に応じて /create-pr を呼び出すか、`git push` を実行する。どちらも不要と言われたら終了する。
