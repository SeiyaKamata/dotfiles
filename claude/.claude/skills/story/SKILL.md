# ユーザーストーリースキル

## 役割
機能要望をPdM・ステークホルダーが読めるユーザーストーリー形式でまとめる。
技術的な詳細は書かず、**ビジネス価値とユーザー体験**に焦点を当てる。

## 出力先
`specs/<feature>/stories.md`

## 進め方

### Step 1: コンテキスト収集
- 既存の `specs/<feature>/stories.md` があれば読み込む
- `specs/<feature>/requirements.md` があれば参照する

### Step 2: ヒアリング
ユーザーの要望を受け取り、以下を質問して詰める：
- **誰が使うか**: ユーザーの役割・属性
- **何をしたいか**: 達成したいゴール
- **なぜ必要か**: ビジネス上の価値・背景
- **完了の基準**: どうなれば「できた」と言えるか

技術的な実装方法は聞かない。

### Step 3: ユーザーストーリー作成
ドラフトを作成してユーザーに確認を取る。

### Step 4: 保存
承認されたら `specs/<feature>/stories.md` に保存する。
`specs/<feature>/` ディレクトリがなければ作成する。

## 出力フォーマット

```markdown
# ユーザーストーリー: [機能名]

## ストーリー

### [ストーリー名]
**As a** [ユーザーの役割],
**I want** [やりたいこと],
**so that** [得られる価値・目的].

**受け入れ条件:**
- [ ] [条件1]
- [ ] [条件2]
- [ ] [条件3]

**優先度:** 高 / 中 / 低
**スコープ:** 今回含む / 将来対応

---

### [ストーリー名2]
...
```

## 完了条件
承認され `specs/<feature>/stories.md` に保存できたら完了。
次に `/spec` を使うと、このストーリーを元に要件を詳細化できる。
