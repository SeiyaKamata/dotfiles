# タスク生成スキル

## 役割
設計ドキュメントから、実装者が1〜3時間で完了できる粒度のタスクリストを生成する。

## 自走モード（`auto` 引数）
`$ARGUMENTS[1]` が `auto` の場合、人間の承認を待たずに進める：
- Step 5 の「ユーザーに確認を取る」をスキップする
- Step 4 のレビューゲート（書き込み前チェックリスト）を通過したら保存し、制御を呼び出し元に返す

引数なしの単体起動では従来どおり承認を待つ。

## 入力
- `.specs/<feature>/requirements.md`
- `.specs/<feature>/design.md`
- `.specs/<feature>/meta.md`（PR タイトル・ブランチ名の素材。無ければ自動生成にフォールバック）

## 出力先
- `.specs/<feature>/tasks.md`
- `.specs/<feature>/qa.md`（受け入れ条件から導出した QA シナリオ。`/qa` が消費する）

## タスク設計の原則
- **粒度**: 1タスク = 1〜3時間で完了できる単一責任の作業
- **独立性**: 可能な限りタスク間の依存を減らす（実装順と stacked PR が素直になる）
- **具体性**: 「何をすれば完了か」が明確であること
- **番号体系**: 大タスク（1., 2., ...＝フェーズ）→ サブタスク（1.1, 1.2, ...）の2階層

## フェーズ = 大タスク = 1 PR（重要な前提）
用語の定義は `claude/CLAUDE.md`「用語集」に従う。
- **大タスク（1., 2., ...）は 1 フェーズ = 1 ブランチ = 1 PR に 1:1 で対応させる**。1 つの大タスクを複数フェーズにまたがせない／1 フェーズに複数の大タスクを詰めない。
- 大タスクが複数ある = stacked PR になる（フェーズを積み上げる運用）。
- 大タスクの粒度は「1 PR として無理なくレビューできる大きさ」を目安にする。大きすぎる場合は**大タスク自体を分割**する（実装工程では分割しない前提なので、ここで適切に割っておく）。

## ブランチ名・PR タイトルルール
命名の形式は `claude/CLAUDE.md`「PR / ブランチ命名」に従う。tasks が**各フェーズ（大タスク）の最終的なブランチ名と PR タイトルを確定して tasks.md に明記**する（impl / create-pr はこれを読む）。

**ブランチ名**:
- 単一フェーズ = `<feature>`、複数フェーズ = `<feature>-p1`, `-p2`, ...（`pN` を `p(N-1)` にスタック）

**PR タイトル**: `【鎌田QA】<pr_title>[：<フェーズ内容>]（PR<N>/<M>）`
- `<pr_title>`: `meta.md` の `pr_title`（例 `[SEC-16005] ATM Auth0移行`）。meta.md が無い／`pr_title` 空なら設計・フェーズ内容から自動生成したタイトル
- `：<フェーズ内容>`: 総フェーズ数 `M≥2`（stacked）のときだけ付ける。単一 PR では付けない
- `（PR<N>/<M>）`: 常に付ける（`M`＝総フェーズ数。単一なら `（PR1/1）`）
- `【鎌田QA】` は固定 prefix

タスク一覧では各大タスク（フェーズ）の見出し直下に、確定した**ブランチ名（base 付き）と PR タイトル**を明記する。

## 進め方

### Step 1: 引数チェック
- `$ARGUMENTS[0]` が未指定なら「使い方: /tasks <feature> [auto]」を表示して終了
- feature 名と auto フラグを確定する（`auto` であれば Step 5 のユーザー確認をスキップ）
- それ以外の値は無視する

### Step 2: コンテキスト収集
- `.specs/<feature>/requirements.md` と `.specs/<feature>/design.md` を読み込む
- `.specs/<feature>/meta.md` があれば読み込み、`pr_title` / `branch_name` / `ticket_key` を PR タイトル・ブランチ名の素材にする（無い／空なら自動生成）
- 既存の `.specs/<feature>/tasks.md` があればマージ対象として読み込む

### Step 3: タスク生成
設計の各コンポーネントを実装タスクにマッピングする：
1. まず大タスク（フェーズ）を決める。フェーズが確定したら**各フェーズのブランチ名と PR タイトルを確定する**（「ブランチ名・PR タイトルルール」に従う。総フェーズ数 `M` が決まるので `（PR N/M）` もここで確定させ、各大タスクの見出し直下に明記する）
2. 各大タスクをサブタスクに分解する
3. 依存関係を確認し、`_Depends:_` に前提タスクを明記して実装順を決める
4. 各サブタスクに「完了条件」を1行で記述する
5. **QA シナリオを導出する** — `requirements.md` の受け入れ条件から `.specs/<feature>/qa.md` の QA シナリオ（`前提 / 手順 / 期待 / _Requirements: N_`）を起こす。**ローカルでブラウザ操作により検証できる受け入れ条件だけ**をシナリオ化する（下記「QA シナリオの範囲」に従う）
6. **他 PJ への handoff を洗い出す** — この feature の実装が**別 PJ の変更を前提とする**場合、それを `## 他 PJ への handoff` に `H<n>` として記録する（下記「他 PJ への handoff の洗い出し」に従う）。ブロックされる実装タスクには `_Depends:_` から `H<n>` を参照させる。

全ての要件（Requirement 1, 2, ...）がいずれかのタスクにカバーされていることを確認する。

#### 他 PJ への handoff の洗い出し（tasks.md に何を書くか）
この feature を完成させるのに、**この作業ディレクトリからは直接編集できない別 PJ の変更**が必要な場合、それは `/impl` で実装できない。tasks.md にコーディングタスクとして混ぜず、`## 他 PJ への handoff` に分けて記録する。`/handoff` がこの項目を消費し、依頼を投げたらチェックを付ける。

- **記録する**: 別 PJ の API/DB スキーマ/共有ライブラリ/画面 などに変更が要る、別 PJ 側で先に対応が終わらないとこの PJ の実装・結合ができない、といった依存。
- **記録しない**: この PJ 内で完結する変更、単なる外部 API の呼び出し（相手 PJ に変更を頼まないもの）。
- **書く内容**: 依頼先 PJ（略称でよい）・何を依頼するか 1 行・理由・どのフェーズ/大タスクをブロックするか。実装手順や受け入れ条件は書かない（依頼先の `/spec` が起こす。`/handoff` 参照）。
- ブロックされる大タスク/サブタスクの `_Depends:_` に `H<n>` を含め、handoff 未完了のまま着手しないようにする。
- 他 PJ への依存が無ければこの見出しは省略してよい。

#### QA シナリオの範囲（qa.md に何を書くか）
`qa.md` は `/qa` が `qa-browser`（Playwright）で**ローカル起動したアプリをブラウザ操作して検証する**ためのもの。ローカルのブラウザ操作で確認できることだけをシナリオにする。**受け入れ条件を無理に全網羅しようとしない**（網羅は tasks.md 側の責務）。

- **書いてよい（ブラウザで観測可能）**: 画面遷移・表示内容・フォーム入力と送信結果・バリデーションエラー表示・一覧/詳細の表示・ボタン活性/非活性・トースト/モーダル・権限による画面差分・UI に反映される状態変化。
- **書かない（ローカルのブラウザでは検証できない）**: メール/SMS/プッシュの実送信・外部 API や決済プロバイダの実通信・Webhook 受信・cron/バッチ/スケジュール実行・本番専用インフラや環境変数依存の挙動・実時間の経過待ち（長時間 TTL 等）・DB を直接見ないと分からない副作用・サードパーティアカウントを要する操作。
- **代替できるなら書く**: 上記「書かない」項目でも、UI 上に結果が現れるなら**その UI 表示を期待値**にする（例: 「確認メール送信」→ 実送信ではなく「送信完了メッセージが画面に出る」ことを確認する）。UI に一切現れないものはシナリオから除外する。
- 除外した受け入れ条件は qa.md 末尾の「## ローカル QA 対象外」に理由付きで箇条書きし、検証手段（手動確認 / CI / 別途）を1行添える。

### Step 4: レビューゲート（書き込み前に実施）
- [ ] 全要件がタスクにマッピングされているか
- [ ] 各タスクに完了条件が書かれているか
- [ ] タスクの順序が論理的か（依存関係が正しいか）
- [ ] サブタスクが適切な粒度か（大きすぎ・小さすぎではないか）
- [ ] 各フェーズ（大タスク）にブランチ名（base 付き）と PR タイトルが明記されているか（形式は `claude/CLAUDE.md`「PR / ブランチ命名」どおりか。`（PR N/M）` の M が総フェーズ数と一致するか）
- [ ] `qa.md` の各シナリオがローカルのブラウザ操作だけで検証できるか（実送信・外部通信・バッチ等を含まないか）
- [ ] ブラウザで検証できない受け入れ条件を `## ローカル QA 対象外` に逃がしてあるか
- [ ] 別 PJ の変更を前提とする依存を `## 他 PJ への handoff` に `H<n>` で洗い出し、ブロックされるタスクの `_Depends:_` から参照してあるか（他 PJ 依存があれば）

### Step 5: ユーザーレビュー
保存済みの `.specs/<feature>/tasks.md` をユーザーに確認してもらう。
修正が入った場合は Step 3 に戻ってタスクを更新し、Step 4 を再実行する。

### Step 6: 次ステップ提示
承認されたら、末尾の「次ステップ提示」の定型ブロックで次の手順を提示する。

## 出力フォーマット

```markdown
# 実装タスク: [機能名]

## タスク一覧

- [ ] 1. [大タスク名]
  - ブランチ: `<feature>-p1`（base: `<デフォルトブランチ>`）
  - PR タイトル: `【鎌田QA】<pr_title>：[フェーズ1の内容]（PR1/2）`
- [ ] 1.1 [サブタスク名]
  - [実装内容の説明]
  - 完了条件: [何をもって完了とするか]
  - _Requirements: 1, 2_

- [ ] 1.2 [サブタスク名]
  - [実装内容の説明]
  - 完了条件: [何をもって完了とするか]
  - _Requirements: 3_

- [ ] 2. [大タスク名]
  - ブランチ: `<feature>-p2`（base: `<feature>-p1`）
  - PR タイトル: `【鎌田QA】<pr_title>：[フェーズ2の内容]（PR2/2）`
- [ ] 2.1 [サブタスク名]
  - _Depends: 1.1, 1.2, H1_
  - 完了条件: [何をもって完了とするか]
  - _Requirements: 4_

## 他 PJ への handoff
- [ ] H1. [依頼先PJ（略称可）] — [何を依頼するか 1 行]
  - 理由: [なぜこの PJ の変更が必要か]
  - ブロック対象: [この handoff が終わらないと進めないフェーズ/大タスク]（例: 大タスク2）
```

`## 他 PJ への handoff`:
- `H<n>`: 連番の handoff ID。ブロックされるタスクの `_Depends:_` から参照する。
- チェックボックス: `/handoff` が依頼書（seed.md）を書いたら `[x]` に更新する（＝依頼済みの印。相手 PJ 側の完了ではなく「投げた」ことを表す）。
- 他 PJ への依存が無ければこのセクションは省略してよい。

### qa.md（`.specs/<feature>/qa.md`）

```markdown
# QAシナリオ: [機能名]

## シナリオ一覧
- [ ] S1. <シナリオタイトル>
  - 前提: <開始状態>
  - 手順: 1.<操作> 2.<操作> 3.<操作>
  - 期待: <ブラウザ画面上で観測できる期待結果>
  - _Requirements: <対応する要件番号>_
- [ ] S2. ...

## ローカル QA 対象外
- <ブラウザで検証できない受け入れ条件>（_Requirements: N_）— 検証手段: <手動確認 / CI / 別途>
```

- `S<n>`: 連番のシナリオ ID。`/qa` が `qa-browser` の結果配列とこの ID で対応づける。
- チェックボックス: `/qa` が合否で `[x]`（pass）/`[ ]`（fail のまま）に更新する。
- 各シナリオは requirements.md の受け入れ条件と対応づける。**ローカルのブラウザ操作で検証できるものだけ**を載せ、期待値は必ず画面上で観測できる事象にする（「QA シナリオの範囲」参照）。
- ブラウザで検証できない受け入れ条件は `## シナリオ一覧` に入れず、`## ローカル QA 対象外` に理由と検証手段付きで記載する。対象外が無ければこの見出しは省略してよい。

## 完了条件
ユーザーが承認し、`.specs/<feature>/tasks.md` と `.specs/<feature>/qa.md` の両方を保存できたら完了。
次は `/impl` を起動する。

## 次ステップ提示
単体起動で完了したら、次の定型ブロックを**コードフェンスで囲まず**プレーンテキストで出力し、次スキルは自動起動せずユーザーの実行を待って終了する。

```
✅ タスク分解完了 — .specs/<feature>/tasks.md（QA シナリオは .specs/<feature>/qa.md）
OK：/impl <feature>
NG：/tasks <feature>
```

自律モード（起動引数に `auto` を含む）では上記ブロックを出さず、遷移先を 1 行の簡易ログだけ残す（例: `次: /impl <feature>`）。次スキルの起動は呼び出し元（orchestrator）が行う。
