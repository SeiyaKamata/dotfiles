# プロトタイプスキル

## 役割
本番と同じFWで、design.md の対象機能を**実際に動く形で作り切る**。モックではない。ロジック・API結線・状態管理も**本物**にする。そこで判明した実物の制約・破綻を `design.md` に書き戻し、設計の精度を高める。

ここでの実装は**品質を問わない捨て実装（spike）**である。可読性・保守性・命名・テストは**一切考えなくてよい**。とにかく動けばいい。

**成果物は 2 つ**：
1. `design.md` への書き戻し（設計の精度向上）＝**仕様の記録**
2. `<feature>-proto` ブランチに残した**動くコード**＝ impl の**流用元**

このコードは捨てない。本番実装（`/impl`）は白紙から書き直すのではなく、この prototype コードを**読んで流用（コピー＋本番品質に整形）**する（＝「参照して昇格」）。ただし仕様の記録（=tasks / impl が正典として参照するもの）はあくまで `design.md`。prototype コードは、それを速く実装するための**参考実装**という位置づけ。

## ブランチ名ルール
ブランチ名は `<feature>-proto` とする。impl はここから流用するので、このブランチとコードは残す。

## いつ使うか
大きい修正で、かつ UI を伴うときだけ使う。小さい修正・バックエンドのみの修正では使わない（確かめる画面がないため）。起動判定は orchestrator が行う。

## 入力
- `.specs/<feature>/requirements.md`
- `.specs/<feature>/design.md`

## 出力先
- `.specs/<feature>/design.md`（作り切って得た気づきを書き戻す＝仕様の記録）
- `<feature>-proto` ブランチにコミットして残す動くコード（impl の流用元）

## 進め方

### Step 1: 引数チェック
- `$ARGUMENTS[0]` が未指定なら「使い方: /prototype <feature> [auto]」を表示して終了
- feature 名を確定する
- 引数のいずれかに `auto` を含む場合は自律モード（ブロック抑制）とする。末尾の次ステップ提示で定型ブロックを出さず、1 行の簡易ログのみ残す。

### Step 2: ブランチ準備とコンテキスト収集
- デフォルトブランチから `<feature>-proto` を作成して切り替える
  ```
  DEFAULT=$(git remote show origin | sed -n 's/.*HEAD branch: //p')
  git checkout "$DEFAULT" && git pull
  git checkout -b <feature>-proto
  ```
- `requirements.md` と `design.md` を読み込む
- design.md の画面構成・データフロー・コンポーネント構成を把握する
- 本番のFW・コンポーネントパターン・API/データ層を Grep/Glob で確認する（本物として動かし、かつ impl が流用しやすくするため）

### Step 3: 本物として作り切る
- 本番と同じFWで、design.md の対象機能を**動く形で最後まで実装する**
- ロジック・API結線・状態管理は**本物**にする。ダミーデータやモック関数で誤魔化さない。要件どおりに動くことをここで確かめる
- ただし**品質は問わない**：可読性・命名・責務分割・テスト・エラーハンドリングの網羅は無視してよい。とにかく動かすことを最優先する
- 本番のデータ・外部APIに触れる必要があるなら、破壊的操作にならない範囲で確認する（書き込み・課金・不可逆操作の前は必ず確認を挟む）

### Step 4: 動かして確認する
- ローカルで起動し、対象機能を要件どおりのフローで通しで動かす
- 実際に作り切ったからこそ判明する点（データ構造の過不足・API のレスポンス形・状態遷移の抜け・想定外の制約）を洗い出す
- 気づいた点をメモする（次の Step 5 で design.md に書き戻す）

> ブラウザでの受け入れ確認（要件どおり画面が動くかの最終判定）は `/qa` の担当。ここでは作り切って得た設計上の気づきを拾うことに専念する。

### Step 5: 気づきを design.md に書き戻す
作り切って判明したことを `design.md` に反映する（仕様の記録の更新）：
- 画面構成・コンポーネント分割の確定
- 画面遷移・操作フローの確定
- データ構造・API インターフェースの確定（実物で判明した形に修正）
- やってみて判明した制約・設計の見直し点

**設計が根本的に破綻していると分かった場合**は、orchestrator に「design への差し戻しが必要」と報告する。

### Step 6: prototype コードを残す
- 動くコードを `<feature>-proto` ブランチにコミットして**残す**。impl がここから流用する
  ```
  git add -A
  git commit -m "proto: <feature> 流用元の動作確認済み捨て実装"
  ```
  > この 1 コミットは PR にならない参照用ブランチへの捨て実装なので、`/commit` スキルは通さず直接コミットしてよい（グローバル規約「必ず /commit を使う」の明示的な例外）。
- prototype コードは正典ではない。impl が正典として参照するのは `design.md`。コードはそれを速く実装するための参考実装。

## 完了ゲート
- [ ] 対象機能がローカルで本物として動き、要件どおりのフローを通せた
- [ ] 作り切って得た気づきを `design.md` に書き戻した
- [ ] 動くコードを `<feature>-proto` ブランチにコミットして残した

## 自走モード（orchestrator 経由）での振る舞い
- 人の目視承認を待たず、作り切って動いたことで設計の妥当性を判断する
- UI の受け入れ確認は `/qa` に委ね、ここでは design.md の精度向上に専念する
- design.md への書き戻しと proto ブランチへのコミットが終わったら次工程（/tasks）へ進む

## 完了条件
design.md への書き戻しと `<feature>-proto` へのコミットが完了したら完了。次は `/tasks` を起動する。

## 次ステップ提示
単体起動で完了したら、次の定型ブロックを**コードフェンスで囲まず**プレーンテキストで出力し、次スキルは自動起動せずユーザーの実行を待って終了する。設計が根本的に破綻していると分かった場合は、ブロックの代わりに design への差し戻しが必要な旨を報告する。

```
────────────────────────────────
✅ 作り切り検証・design 反映・proto コード保存 完了
📄 .specs/<feature>/design.md（更新）
🌿 <feature>-proto（流用元コードを保存）
▶ 次のステップ
   /tasks <feature>
   理由: 設計の精度が上がったので実装タスクに分解する
────────────────────────────────
```

自律モード（起動引数に `auto` を含む）では上記ブロックを出さず、遷移先を 1 行の簡易ログだけ残す（例: `次: /tasks <feature>`）。次スキルの起動は呼び出し元（orchestrator）が行う。
