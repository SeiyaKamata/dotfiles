# プロトタイプスキル

## 役割
本番と同じFWでローカルに動くUIモックを作り、ブラウザで操作感を確かめる。そこで得た気づきを `design.md` に書き戻し、設計の精度を高める。

モックは**ローカル限定の使い捨て**であり、git に入れない。資産として残るのは `design.md` への書き戻しだけ。tasks / impl が参照するのは `design.md` であってモックコードではない。

## ブランチ名ルール
ブランチ名は `<feature>-proto` とする。

## いつ使うか
大きい修正で、かつ UI を伴うときだけ使う。小さい修正・バックエンドのみの修正では使わない（確かめる画面がないため）。起動判定は orchestrator が行う。

## 入力
- `.specs/<feature>/requirements.md`
- `.specs/<feature>/design.md`

## 出力先
- `.specs/<feature>/design.md`（モックで得た気づきを書き戻す）
- モックコード本体は git に入れない（`.gitignore` 配下のスクラッチ、または確認後に破棄する作業ツリー）

## 進め方

### Step 1: 引数チェック
- `$ARGUMENTS[0]` が未指定なら「使い方: /prototype <feature> [auto]」を表示して終了
- feature 名を確定する
- 引数のいずれかに `auto` を含む場合は自律モード（ブロック抑制）とする。末尾の次ステップ提示で定型ブロックを出さず、1 行の簡易ログのみ残す。

### Step 2: コンテキスト収集
- `requirements.md` と `design.md` を読み込む
- design.md の画面構成・データフロー・コンポーネント構成を把握する
- 本番のFW・コンポーネントパターンを Grep/Glob で確認する（モックを本番に近づけるため）

### Step 3: モックを実装する
- 本番と同じFWで、design.md の主要画面を動く形で実装する
- ロジック・API結線・状態管理は本物でなくてよい（ダミーデータ・モック関数で代替）。確かめたいのは**画面構成と操作フロー**
- モックは git に入れない。誤コミットを防ぐため、置き場は `.gitignore` 配下のスクラッチを優先する

### Step 4: モックを操作して確認する
- ローカルで起動したモックの主要画面を遷移・操作する
- 操作フローが破綻していないか、design.md の想定と実物がズレていないかを確認する
- 気づいた点をメモする（次の Step 5 で design.md に書き戻す）

> ブラウザでの動作確認そのもの（要件どおり画面が動くかの受け入れ確認）は `/qa` の担当。ここではモックの操作感と設計の妥当性だけを見る。

### Step 5: 気づきを design.md に書き戻す
モックを動かして判明したことを `design.md` に反映する。ここが本スキルの成果物：
- 画面構成・コンポーネント分割の確定
- 画面遷移・操作フローの確定
- やってみて判明した制約・設計の見直し点

**設計が根本的に破綻していると分かった場合**は、orchestrator に「design への差し戻しが必要」と報告する。

### Step 6: モックコードの後始末
- モックコードは commit しない。スクラッチに置いたものはそのまま残しても git に入らない。作業ツリーに書いた場合は破棄する

## 完了ゲート
- [ ] 主要画面がローカルで動き、操作確認できた
- [ ] モックで得た気づきを `design.md` に書き戻した
- [ ] モックコードを git に入れていない

## 自走モード（orchestrator 経由）での振る舞い
- 人の目視承認を待たず、モックの操作確認で設計の妥当性を判断する
- UI の受け入れ確認は `/qa` に委ね、ここでは design.md の精度向上に専念する
- design.md への書き戻しが終わったら次工程（/tasks）へ進む

## 完了条件
design.md への書き戻しが完了したら完了。次は `/tasks` を起動する。

## 次ステップ提示
単体起動で完了したら、次の定型ブロックを**コードフェンスで囲まず**プレーンテキストで出力し、次スキルは自動起動せずユーザーの実行を待って終了する。設計が根本的に破綻していると分かった場合は、ブロックの代わりに design への差し戻しが必要な旨を報告する。

```
────────────────────────────────
✅ モック検証と design 反映完了
📄 .specs/<feature>/design.md（更新）
▶ 次のステップ
   /tasks <feature>
   理由: 設計の精度が上がったので実装タスクに分解する
────────────────────────────────
```

自律モード（起動引数に `auto` を含む）では上記ブロックを出さず、遷移先を 1 行の簡易ログだけ残す（例: `次: /tasks <feature>`）。次スキルの起動は呼び出し元（orchestrator）が行う。
