# 変更管理スキル

## 役割
実装中に仕様・設計・タスクの変更が必要になったとき、影響範囲を判断して
適切な工程に戻し、下流のドキュメントを更新する。

## 変更レベルの判断基準

### 要件レベル（最上流）
以下に当てはまる場合：
- ユーザーから見える振る舞いが変わる
- スコープの追加・削除がある
- 「何を作るか」が変わる

→ `.specs/<feature>/requirements.md` を更新 → `/design` からやり直し

### 設計レベル
以下に当てはまる場合：
- 要件は変わらないが、実現方法が変わる
- コンポーネント構成・データ構造・APIが変わる
- 「どう作るか」が変わる

→ `.specs/<feature>/design.md` を更新 → `/tasks` からやり直し

### タスクレベル（最下流）
以下に当てはまる場合：
- 要件も設計も変わらないが、実装の分割・順序が変わる
- タスクの追加・削除・統合が必要
- 「何から作るか」が変わる

→ `.specs/<feature>/tasks.md` を更新 → `/impl` に戻る

## 進め方

### Step 1: 引数チェック
- `$ARGUMENTS[0]` が未指定なら「使い方: /change <feature>」を表示して終了
- feature 名を確定する

### Step 2: 変更内容をヒアリング
ユーザーから変更内容を受け取る。
不明な点があれば質問して変更レベルを確定させる。

### Step 3: 影響範囲の確認
現在の `.specs/<feature>/requirements.md`, `.specs/<feature>/design.md`, `.specs/<feature>/tasks.md` を読み込み、
変更がどのドキュメントに影響するかを確認する。

### Step 4: 変更レベルを提示して確認
判断した変更レベルとその理由をユーザーに提示し、承認を得る。

例：
> この変更は**設計レベル**です。
> 理由：要件（何を作るか）は変わらず、データ構造の変更のみのため。
> `.specs/<feature>/design.md` を更新して `/tasks` からやり直します。よろしいですか？

### Step 5: ドキュメントを更新
承認後、該当レベルのドキュメントを更新する。
**下流のドキュメントも影響を受ける場合は合わせて更新する。**

| 変更レベル | 更新するドキュメント |
|------------|---------------------|
| 要件 | requirements.md → design.md → tasks.md |
| 設計 | design.md → tasks.md |
| タスク | tasks.md のみ |

### Step 6: 次ステップ提示
更新完了後、末尾の「次ステップ提示」の分岐ブロックで、確定した変更レベルに対応する再開スキルを提示する。

## 完了条件
ドキュメントの更新が完了し、次のスキルへ誘導できたら完了。

## 次ステップ提示
完了したら、確定した変更レベルに対応する次の分岐ブロックを**コードフェンスで囲まず**プレーンテキストで出力し、次スキルは自動起動せずユーザーの実行を待って終了する。Step 4 で変更レベルが確定しているので、該当する 1 つを提示する（`📄` は実際に更新した md のパスにする）。

```
✅ ドキュメント更新完了 — <更新した md（requirements.md / design.md / tasks.md）>
OK（要件レベル）：/design <feature>
OK（設計レベル）：/tasks <feature>
OK（タスクレベル）：/impl <feature>
```
